# Дизайн: агент печати + счётчик (под-проект №2)

Дата: 2026-07-02
Статус: утверждён (брейншторм), готов к декомпозиции на планы

## Контекст

Мультитенантное ядро (под-проект №1) готово: бот принимает заказы (фото + формат
+ crop-данные), админка/студии/роли/webhook-мультибот работают, таблица
`billing_events` заведена (Плана 1), но НЕ наполняется. Печать сейчас ручная.

Цель №2 — локальный **агент печати** на ПК студии, который: тянет
подтверждённые заказы, применяет обрезку, собирает фото в макеты, печатает на
принтере (оператор подтверждает) и рапортует факт печати на сервер, наполняя
`billing_events` — это источник комиссии Ивана (5₽ × напечатанное фото), поверх
которого План 3 строит счета студиям.

Текущий поток студии (со слов владельца): Windows-ПК с подключённым
фотопринтером; фото прогоняются через Photoshop под макеты, печать — через
встроенное приложение «Фотографии» (там же размер бумаги/обрезка). Агент
автоматизирует crop + сборку макетов + печать.

## Утверждённые решения (брейншторм 2026-07-02)

- **Среда:** Windows-ПК студии + фотопринтер, печать через системный спулер.
- **Связь агент↔сервер:** pull — агент периодически опрашивает API (работает за
  NAT без публичного IP, как ранее polling бота).
- **Автоматизм:** оператор подтверждает печать (не полный авто-принт) — меньше
  риск брака/лишних отпечатков.
- **Рендер:** агент применяет crop (из БД) и **автоматически собирает фото в
  макеты** на ПК (напр. инстакс — несколько фото на листе рядами). Сервер про
  раскладку не знает.
- **Макеты — на агенте:** раскладка привязана к принтеру/бумаге, а не к
  бизнес-каталогу → хранится в локальном конфиге агента (формат→макет) с
  редактируемыми дефолтами в комплекте. Схему `Product` НЕ трогаем.
- **Биллинговая единица:** напечатанное **фото (позиция)**, не лист. Событие
  создаёт сервер на `/api/print/report` (по факту печати; брак/перепечатка —
  вне тарификации, оператор не рапортует брак).
- **v1: один агент на студию.** Мультиагентность в одной студии и координация
  (claiming/локи заданий против двойной печати) — отложено. v1 предполагает один
  активный агент; модель `PrintAgent` допускает несколько пайрингов на будущее,
  но без межагентной координации.

## Архитектура

### 1. Сервер — Print API (аутентификация агента)

- **Пайринг:** админка (studio_admin своей студии или super_admin) генерирует
  одноразовый код привязки для студии. Агент меняет код на долгоживущий
  **токен агента** (привязан к `studio_id`). Токен хранится хешем.
- **Модель `PrintAgent`:** `id, studio_id (FK), name, token_hash, pairing_code
  (nullable, до активации), paired_at, last_seen_at, printer_status (str),
  queue_len (int)`. Несколько строк на студию допустимо (на будущее); v1
  использует один активный агент.
- **Эндпоинты (требуют токен агента, resolve → studio_id):**
  - `POST /api/print/pair` — тело `{code}` → возвращает токен агента (обмен кода).
  - `GET /api/print/jobs` — подтверждённые (CONFIRMED) заказы студии, ещё не
    напечатанные: `[{order_number, order_id, photos: [{photo_id, product_slug,
    aspect_ratio, crop_data}]}]`.
  - `GET /api/print/photo/{photo_id}` — байты оригинала (сервер тянет из Telegram
    по расшифрованному токену бота студии; photo_id обязан принадлежать студии
    агента — иначе 404). Crop применяет агент.
  - `POST /api/print/report` — тело `{order_id, printed_photo_ids: [...]}` →
    сервер проверяет принадлежность заказа студии, создаёт `billing_event` на
    каждое фото (fee = studio.platform_fee_per_photo, printed_at=now),
    идемпотентно (повторный рапорт того же фото не дублирует событие), двигает
    статус заказа (CONFIRMED→PRINTING→READY по завершении).
  - `POST /api/print/health` — `{printer_status, queue_len}` → обновляет
    PrintAgent.last_seen_at/printer_status/queue_len; супер-админ видит в /studios.

### 2. Агент (Windows, Python, упаковка в .exe)

- **Пайринг** при первом запуске: оператор вводит код из админки → агент
  получает и хранит токен локально (защищённо).
- **Цикл pull:** периодически `GET /api/print/jobs`; для новых заказов —
  скачивает фото (`/api/print/photo/{id}`), применяет crop (Pillow), собирает в
  листы по локальному конфигу макетов (формат-slug→раскладка: размер листа мм,
  ряды×колонки, отступы, DPI, ориентация; дефолты для полароид/инстакс/10×15).
- **Очередь-UI:** оператор видит заказы, превью собранных листов, жмёт «Печать».
- **Печать:** собранные листы → выбранный Windows-принтер через спулер
  (pywin32/GDI или ShellExecute «print»); размер/бордюры — в настройках принтера
  по умолчанию. После успешной отправки — `POST /api/print/report` с id
  напечатанных фото → биллинг; заказ помечается печатаемым/готовым.
- **Heartbeat:** периодический `POST /api/print/health`.
- **Конфиг макетов:** локальный файл (YAML/JSON) с дефолтами, легко правится.

### 3. Биллинг

`billing_event` (studio_id, order_id, photo_position/photo_id, fee, printed_at)
наполняется сервером на `/api/print/report`. План 3 (месячные счета студиям +
kill-switch за неуплату) агрегирует эти события.

## Декомпозиция №2 (по тестируемости)

- **2-I — Серверный Print API + пайринг + report/billing + health.** Чистая
  серверная работа; полностью тестируется в текущем pytest-стенде, без Windows.
  *(первый спек→план→реализация)*
- **2-II — Ядро агента:** пайринг-клиент, pull заданий, движок crop+сборки
  макетов (Pillow), локальный конфиг макетов с дефолтами. Компоновка — чистый
  Pillow, тестируется на любой ОС; Windows не нужен.
- **2-III — Печать на Windows** (спулер) + очередь-UI + упаковка .exe +
  heartbeat-интеграция. Требует реальной Windows-машины студии для валидации.

Порядок: 2-I → 2-II → 2-III. 2-I и 2-II строятся/тестируются сейчас; 2-III —
хардварно-зависимая часть, доводится на машине студии.

## Изоляция (инвариант, как в ядре)

Каждый эндпоинт Print API резолвит `studio_id` из токена агента и фильтрует по
нему: агент студии A не может получить jobs/photo/report чужой студии. `photo_id`
в `/photo` и `order_id` в `/report` проверяются на принадлежность студии агента
(иначе 404). Это красная нить тестов 2-I.

## Вне рамок №2

- Мультиагентность в одной студии + координация заданий (claiming/локи).
- План 3: месячные счета студиям, kill-switch за неуплату, онбординг-визард.
- Реальная сборка макетов в PS (заменяется crop ботом + авто-раскладкой агента).
