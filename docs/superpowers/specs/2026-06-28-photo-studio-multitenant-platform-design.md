# Дизайн: мультитенантная платформа для фотостудий

Дата: 2026-06-28
Статус: утверждён (брейншторм), готов к декомпозиции на планы реализации

## Контекст

Текущий сервис (`photo28`) — монолит под одну фотостудию (сестры):
Telegram-бот (aiogram, polling) принимает заказ на печать фото в 8 шагов,
FastAPI-админка ведёт заказы/товары/промокоды/настройки, оплата — ручной P2P
(реквизиты в боте → клиент переводит → шлёт фото-чек → менеджер подтверждает),
печать — ручная (менеджер скачивает фото из админки и печатает сам). Всё
захардкожено под одного владельца: один токен бота, один чат менеджеров, одни
реквизиты, один Яндекс.Диск, одна SQLite-база. Понятия «студия/тенант» нет.

**Прод ещё не запускался, данных нет** — проектируем с чистого листа, миграция
старой базы не требуется, сестра становится первой студией через общий онбординг.

## Цель

Превратить сервис в мультитенантную платформу, которую владелец (Иван) продаёт
другим фотостудиям как SaaS: каждой студии — свой Telegram-бот, своя группа для
чеков, свой локальный софт печати, своя админка. Иван админит все студии из
супер-админки и зарабатывает на комиссии за печать.

## Бизнес-модель (утверждена)

- **Монетизация:** постоплата по счётчику (вариант B). Деньги клиентов идут
  напрямую студии (P2P или собственный шлюз студии) — Иван **не оператор чужих
  денег**, что снимает агентскую схему / 54-ФЗ / возвраты с него.
- **Комиссия:** `platform_fee_per_photo` (по умолчанию 5 ₽) × число **успешно
  напечатанных** фото. Раз в месяц Иван выставляет студии счёт. Это его выручка
  как поставщика ПО (проводит сам как самозанятый/ИП).
- **Биллинговое событие — факт печати**, а не подтверждение оплаты: брак,
  замятие, отменённые/возвращённые заказы **не тарифицируются**. Перепечатка
  брака не создаёт нового события.
- **Биллинговая единица — напечатанное изображение (позиция)**, а не заказ и не
  строка прайса. Заказ из 28 полароидов = 28 событий, а не 1 (важно из-за
  тиерного/группового ценообразования в прайсе).
- **Рычаг взыскания:** бот и агент печати работают на инфраструктуре Ивана.
  Неуплата месячного счёта → `is_active=false` → студия отключается (kill-switch).
- **Цена 5 ₽:** для студии незаметна (~6–12 % их маржи при отпечатке 50–100 ₽,
  себестоимость 5–15 ₽) — легко продаётся. Месячного минимума по решению
  владельца нет; поле `monthly_minimum` остаётся в схеме со значением 0
  (выключено) на будущее.
- **GTM-рычаг:** шаблон каталога типовых форматов при создании студии (владелец
  только правит цены). Бесплатный месяц / setup-fee — не вводим.

## Утверждённые архитектурные решения

| Аспект | Решение |
|---|---|
| Сбор комиссии | Постоплата по счётчику; событие — по факту печати (агент печати) |
| Оплата клиента | Оба режима: ручной P2P по умолчанию + опциональный собственный шлюз студии |
| Запуск ботов | Один процесс, много ботов, **webhook**, маршрут по токену в URL |
| Админка | Одно FastAPI-приложение, роли super-admin / studio-admin |
| БД | PostgreSQL с чистого листа, общая база, изоляция через `studio_id` |
| Данные изоляции | `users` уникальны по (studio_id, telegram_id); своя нумерация заказов на студию |

## Декомпозиция на под-проекты

Каждый под-проект проходит свой цикл спек → план → реализация.

1. **Мультитенантное ядро** *(фундамент — детальный спек ниже)*
2. **Агент печати + счётчик** — локальный Windows-агент: пайринг по коду, печать
   на принтере, `billing_event` при успешном отпечатке, health-отчёт в
   супер-админку (онлайн / принтер готов / очередь).
3. **Биллинг + онбординг-визард** — месячные счета студиям, kill-switch за
   неуплату, шаблон каталога, 5-шаговый визард самообслуживания.
4. **Сайт-лендинг** — описание сервиса, цена, «как работает», форма заявки.

Порядок важен: №2 поднят в фундамент, т.к. факт печати — единственный честный
источник биллинга и одновременно главная ценность/нагрузка по поддержке.

## Воронка продажи и онбординга (целевой сценарий)

```
Сайт-лендинг (цена, как работает) → владелец пишет Ивану → Иван в супер-админке
создаёт студию (1 клик) и шлёт ссылку на визард → владелец проходит 5 шагов:
  1. вставить токен бота (создаёт его сам в @BotFather — программно нельзя)
  2. привязать группу чеков (бот ловит chat_id сам по команде /bind в группе)
  3. установить и спарить агент печати (код пайринга из админки)
  4. подтвердить шаблон каталога, поправить цены
  5. ввести реквизиты P2P (или подключить свой платёжный шлюз)
→ студия в эфире
```

Ручная часть Ивана — один разговор + один клик «создать студию».

## Детальный спек под-проекта №1: Мультитенантное ядро

### Модель данных

Новая таблица `studios`:
```
studios:
  id, slug, name, is_active (kill-switch),
  bot_token (Fernet-шифр), bot_username,
  manager_chat_id (группа чеков), manager_username,
  payment_phone, payment_card, payment_receiver,
  payment_gateway_creds (Fernet-шифр, опц.),
  yandex_disk_token (Fernet-шифр, опц.),
  platform_fee_per_photo (default 5.00),
  monthly_minimum (default 0 — выключено),
  created_at
```

- `studio_id` (FK, NOT NULL, индекс) добавляется во все доменные таблицы:
  `users`, `orders`, `products`, `promocodes`, `settings`.
- `users` уникальны по паре **(studio_id, telegram_id)** — один человек может
  быть клиентом нескольких студий.
- **Нумерация заказов — своя на студию** (студия не должна видеть «дыры» в
  номерах и оценивать обороты других).
- Пути хранения фото: `storage/{studio_id}/{order_number}/...` — против коллизий.
- Промокоды уникальны в рамках студии; кеш настроек — ключ `{studio_id}:{key}`.

Новая таблица `admin_users` (заменяет логин/пароль из `.env`):
```
admin_users:
  id, username, password_hash (bcrypt),
  role (super_admin | studio_admin),
  studio_id (NULL для super_admin)
```

Задел `billing_events` (наполняется под-проектом №2):
```
billing_events:
  id, studio_id, order_id, photo_position,
  fee (snapshot platform_fee_per_photo), printed_at
```

### Конфигурация

Глобальные настройки (`payment_*`, `manager_*`, токены) переезжают из `.env` и
общего кеша `SettingsService` в запись студии и в studio-scoped настройки
(ключ кеша `{studio_id}:{key}`). В `.env` остаётся только инфраструктура:
`DATABASE_URL`, `FERNET_KEY` (шифрование токенов в БД), `SESSION_SECRET`.
Это и есть механизм «подключить студии своего бота, группу чеков и реквизиты» —
всё через супер-админку, без правки кода.

### Bot runtime (webhook, мульти-бот)

- FastAPI принимает апдейты на `/webhook/{token}`; по токену определяется студия,
  `studio_id` кладётся в контекст хендлеров; сервисы фильтруют по нему.
- Один общий `Dispatcher`, множество `Bot`-инстансов. **Per-bot изоляция
  ошибок**: битый/отозванный токен не валит обработку остальных.
- Создание/отключение студии регистрирует/снимает вебхук в Telegram без
  рестарта процесса.
- Приём вебхуков ботов и админка могут жить в одном FastAPI-приложении.

### Админка с ролями

- Middleware достаёт роль и `studio_id` из сессии.
- **studio_admin** жёстко зажат своим `studio_id`: заказы, товары, промокоды,
  настройки, аналитика — только своей студии.
- **super_admin** (Иван): список студий, создание/провижининг студии (токен,
  чат, реквизиты), «смотреть как студия», сводные счётчики по всем, kill-switch.

### Безопасность

- Пароли админов — bcrypt (сейчас хранятся открытым текстом).
- Токены ботов и Я.Диска — Fernet-шифрование в БД, ключ в `.env`.
- Оферта + политика конфиденциальности: **студия — оператор ПДн, Иван —
  обработчик** (через сервер текут фото/телефоны/адреса клиентов чужих студий;
  152-ФЗ).
- Ротация любых секретов, ранее попавших в git-историю.

### Тестирование

Главный риск мультитенантности — **протечка данных между студиями**. Ключевые
тесты:
- Изоляция: studio_admin студии A не видит и не меняет данные студии B.
- Маршрутизация: вебхук с токеном студии A создаёт заказ с `studio_id = A`.
- Ролевой доступ: studio_admin не имеет доступа к супер-админским эндпоинтам.
- Начисление `billing_event` (задел) с корректным `studio_id` и `fee`.

## Вне рамок ядра (передаётся в №2–№4)

- Реальная печать на принтере и health-мониторинг агента — №2.
- Создание `billing_event` при печати, месячные счета, авто-kill-switch — №2/№3.
- Платёжный шлюз студии (фактическая интеграция приёма автоплатежей) — №3.
- Онбординг-визард и шаблон каталога — №3.
- Сайт-лендинг — №4.
